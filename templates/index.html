Discovery Brief Agent

Trigger: HubSpot webhook (meeting booked)  ·  Model: Claude Opus 4.6  ·  lgoldfarb-hub/discovery-brief-agent

Automatically researches a prospect before a discovery call. When a new meeting is booked in HubSpot, this agent pulls deal and contact data, enriches it via Amplemarket, runs live web research, and generates a 16-section MEDDPICC-style brief saved directly as a HubSpot note on the deal. It also advances the deal stage to "First Meeting Scheduled" so the pipeline stays accurate. The goal is for the AE to walk into every call knowing the prospect's integration landscape, pain points, and ICP fit — without doing any manual research.

Rules
🚦 Qualification gate
Only runs for deals in the Global Embedded Pipeline at the "Lead or Opportunity (Meeting Booked)" stage. Any other pipeline or stage → agent exits early with no brief generated. This is the most common reason a brief silently doesn't appear.
🔍 Signal Research reuse
Before running web research for §9 (Trigger Signals), the agent checks whether a Signal Research Agent note already exists on the deal. If found, it reuses that content directly — skipping the web search step entirely. If no signal note exists, it runs run_signal_research() to generate one fresh.

Rules
✅ COMPLETED
Sufficient evidence the intended action was fully executed. "Follow up with X" → any email, call, or note logged today. "Book a meeting" → meeting scheduled or logged. "Send proposal" → email logged or note confirms sent. Compound-intent tasks (e.g. "Follow up with X; schedule a call") require BOTH intents satisfied — an email alone is not enough if scheduling is required.
⚠️ INCOMPLETE
Some activity exists but doesn't fully meet the task intent — e.g. a note was added but no actual outreach was sent, or an email was sent but its content doesn't match the task intent.
❌ MISSED
No engagement of any kind logged today on the deal, and no next task exists to indicate follow-up was rescheduled.
❓ NOT ENOUGH DATA
Task title is too ambiguous to determine intent, deal association is missing, or the action likely happened outside HubSpot (e.g. WhatsApp, phone).
❌ DEALS WITH NO PLANNED TASKS
A separate section appended to each channel's report: all active deals in the tracked stages that have no open task with a future due date. Runs even on days with zero overdue tasks.
🚫 Excluded from evidence
Agent-generated notes are excluded (case-insensitive prefix match): any note starting with "buying intent / intent levels (ai)", "weekly review (ai)", "ai meeting evaluation", "auto-generated", or "reactivation review note:". Content of emails and notes is passed to Claude to verify the engagement actually matches the task intent — activity count alone is not sufficient for Completed.

Rule 2 — Advance to "2nd Meeting Booked"
Deal has 2+ meetings, the latest meeting is in Scheduled or Rescheduled status, and at least one prior meeting was completed → advance stage to "2nd Meeting Booked".
Rule 3 — Mark follow-up completed
Deal has 2+ meetings and all meetings are in Completed status → set status field to "Follow-up meeting completed".
Rule 4 — Post-Meeting Qualification + Business Fit extraction
Deal has exactly 1 meeting in Completed status → advance stage to "Post-Meeting Qualification". Claude reads the Sybill meeting summary note (auto-posted to HubSpot by Sybill after the call) and extracts structured Business Fit fields: buying intent, BANT, integration bottleneck flags, competitor iPaaS in use or being evaluated. Rule 6 — Backfill contact engagements
For every deal (including "Contact later" stage), finds linked contacts and copies their recent engagement records (last 2 weeks) as deal associations — ensuring emails and calls from contacts appear on the deal timeline even if not logged directly against it.
Rule 9 — Reschedule early tasks to 10am CET
For every active-stage deal, fetches all open future tasks. Any task with a due time before 09:00 UTC (10am CET) is rescheduled to 09:00 UTC on the same day — ensuring tasks don't surface before the working day starts.

For every active deal it computes a score out of 100 based on ICP fit, buying intent, and deal value, then uses Claude to generate one specific action per deal. Buying Intent — 50 pts
Primary source: the most recent Buying Intent / Intent Levels (AI) note on the deal (created weekly by HubSpot Enrichment). Second Brain

Trigger: Polling every 30 min  ·  Model: Claude Sonnet 4.6  ·  Hosted on Railway (24/7)  ·  lgoldfarb-hub/second-brain

Leo's personal AI assistant with two interfaces: a Telegram bot (24/7 on Railway) and a local web dashboard (Flask on localhost:5055). Aggregates messages and events from Slack, Pachka, Telegram, Gmail, Front, and Google Calendar into a structured digest every 30 minutes. How it works

Poller (every 30 min): A background thread fetches fresh data from all connectors, upserts items into SQLite (deduped by item ID), then calls Claude to generate a structured digest. The digest is stored in the DB and served to the bot on demand.

Digest structure: Claude classifies items into 4 buckets — (1) Tasks: max 7, things Leo personally must act on (reply, decide, approve) — excludes internal Albato updates, automated notifications, OOO replies, support tickets, things addressed to other teammates; (2) Hot topics: max 4, important things happening at Albato that Leo should know but don't require action — includes deals going Lost or Contact Later; (3) Mentions: max 8, direct messages from real humans waiting for Leo's personal response — excludes anything already in tasks; (4) Meetings: next 48 hours only. Every item includes its original creation date (not fetch time).

Telegram bot: Leo interacts via Telegram. Sending "новости" / "what's new" triggers an immediate poll + returns fetch counts per source (how many items fetched, lookback window used) + today's digest. Any other message uses the DB from the last automatic poll (max 30 min stale). Bot can also take actions: send emails, create calendar events, fetch live items from a specific source with a custom lookback.

News trigger flow: On "новости", bot runs run_once_with_counts(), waits up to 120s, then appends a fetch summary to the user message before passing to Claude: sources, counts, lookback windows, and digest totals (tasks/topics/mentions/meetings).

Telethon (personal Telegram reading): Uses a StringSession stored as TELEGRAM_SESSION_STRING env var — no interactive phone auth needed on Railway. Runs in an isolated thread with its own asyncio event loop to avoid conflicts with python-telegram-bot's async context. Thread replies in Pachka are fetched inline. Slack thread replies fetched via conversations.replies for messages with reply_count > 0.

Google OAuth: credentials.json and token.json are written to /app/ on startup from GOOGLE_CREDENTIALS / GOOGLE_TOKEN env vars (raw JSON). Token auto-refreshes via refresh_token. If token expires permanently, it must be refreshed locally and GOOGLE_TOKEN updated in Railway.
